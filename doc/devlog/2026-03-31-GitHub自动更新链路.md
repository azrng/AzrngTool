# 2026-03-31 GitHub 自动更新链路

## 本次目标
- 通过 GitHub Actions 生成可下载的 Windows 发布 zip
- 在关于页增加检查更新与自动更新入口
- 让客户端可以基于 GitHub Release 完成版本检查与自动替换

## 核心改动
- 新增 `IAppUpdateService` / `AppUpdateService`，调用 GitHub Releases API 获取最新版本与 zip 下载地址
- 新增更新包下载、解压、生成本地 PowerShell 更新脚本的流程，应用退出后自动覆盖当前目录并重启
- 扩展 `IAppInfoService` / `AppInfoService`，统一提供仓库地址、更新包名、安装目录与当前可执行文件路径
- 新增 `IApplicationRuntimeService` / `ApplicationRuntimeService`，通过服务抽象关闭应用，避免在 ViewModel 中直接处理窗口生命周期
- 改造关于页 `AboutPageViewModel` 与 `AboutPageView`，展示当前版本、最新版本、发布时间、更新状态与发布说明，并增加“检查更新 / 立即更新”按钮
- 新增 `.github/workflows/release.yml`，在推送 `v*` tag 时自动执行 `dotnet publish`、压缩 `AzrngTools-win-x64-portable.zip`、上传 artifact 并发布 GitHub Release
- 更新 `AzrngTools/readme.md`，补充 GitHub Actions 发布流程与关于页自动更新说明

## 风险或遗留项
- 首次使用检查更新前，需要仓库已经存在至少一个 GitHub Release；否则客户端会收到 GitHub API 的失败提示
- 自动更新依赖当前安装目录具备写权限；如果程序安装在受保护目录，自动覆盖可能失败，此时需要手动下载 release zip
- 当前更新流程采用“覆盖当前目录”的便携版策略，不会清理历史版本遗留文件；若未来发布结构变化较大，可再补更严格的镜像同步或独立 updater

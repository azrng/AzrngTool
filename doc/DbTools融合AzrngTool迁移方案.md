# DbTools 融合 AzrngTool 迁移清单与信息架构方案

## 1. 目标定位

以 `DbTools` 作为主产品，`AzrngTool` 作为辅助能力来源库。

新的产品定位建议为：

- 面向数据库开发、结构浏览、查询分析、导出文档的一体化桌面工具
- 保持数据库工作流为核心主线
- 将通用工具能力以“开发辅助”形式融入主产品，而不是继续维持“万能工具箱”定位

本方案采用 `方案 B`：

- 左侧导航保留“开发辅助”
- 在“开发辅助”下挂少量高频子项
- 低频工具收纳到页面内的二级切换或分组区块

## 2. 总体原则

- 以 `DbTools` 现有架构和视觉体系为主，不反向迁就 `AzrngTool`
- 优先迁移与数据库开发高相关、高频、低耦合的工具
- 不直接整体搬运 `AzrngTool` 的主窗口、导航和页面壳层
- 迁移时优先复用算法、服务和 ViewModel 思路，UI 重新贴合 `DbTools`
- 每一批迁移都要能独立交付，而不是一次性大合并

## 3. 迁移分层

### 3.1 主体保留在 DbTools

以下能力继续作为主产品核心：

- 连接管理
- 数据库树浏览
- 表 / 视图 / 存储过程详情
- SQL 查询与结果查看
- 导出与文档生成
- 备注编辑与数据库对象维护

### 3.2 从 AzrngTool 抽取辅助能力

将 `AzrngTool` 定位为能力来源库，主要抽取：

- 文本转换能力
- 编解码能力
- 摘要签名能力
- 时间辅助能力
- 轻量结构转换能力

## 4. 迁移清单

### 4.1 第一批，优先迁移

这批建议作为第一阶段落地，优先进入 `DbTools`：

- `JSON 工具`
  - 格式化
  - 压缩
  - 转义 / 去转义
- `SQL 格式化`
- `Base64`
- `URL 编解码`
- `Unicode 编解码`
- `Hash`
- `HMAC`
- `时间戳`
- `GUID`
- `文本压缩 / 文本清洗`

选择理由：

- 与数据库开发日常最相关
- 在查询、接口联调、结构导出、字段处理场景中高频出现
- 迁移成本相对可控
- 可以先验证“开发辅助”区是否真的提升主产品价值

### 4.2 第二批，条件成熟后迁移

- `JWT 解析`
- `Hex 编解码`
- `Gzip 编解码`
- `Json To C#`
- `Json Schema`
- `Markdown 预览`
- `MIME 查询`

选择理由：

- 有一定数据库或接口开发辅助价值
- 但优先级低于第一批
- 更适合作为“开发辅助”扩展能力逐步补充

### 4.3 暂缓迁移

- `AES / DES / RSA / SM4`
- `密码生成器`
- `人民币大写转换`
- `繁简转换`
- `翻译`
- `Base64 转图片`
- `硬件信息`

原因：

- 与数据库主线弱相关
- 容易让产品边界重新发散
- 会稀释 `DbTools` 的核心价值感知

## 5. 信息架构方案

### 5.1 一级导航

建议保留 5 个一级区：

1. `连接与浏览`
2. `查询与分析`
3. `导出与文档`
4. `开发辅助`
5. `设置与关于`

### 5.2 二级结构，采用方案 B

`开发辅助` 下建议保留少量高频子项：

1. `JSON`
2. `编码解码`
3. `签名摘要`
4. `时间工具`
5. `结构转换`

低频工具不继续在左侧展开过深，而是在对应页面内做分组切换。

## 6. 页面结构建议

### 6.1 JSON

页面能力：

- JSON 格式化
- JSON 压缩
- 转义 / 去转义
- 错误提示与定位

页面角色：

- 作为最常用的开发辅助页之一，建议单独二级入口

### 6.2 编码解码

页面内分组：

- Base64
- URL
- Unicode
- Hex
- Gzip

页面角色：

- 聚合型页面
- 左侧二级只保留一个入口，页面内部用 Tab、分段按钮或卡片切换

### 6.3 签名摘要

页面内分组：

- Hash
- HMAC
- JWT 解析，可后续补入

页面角色：

- 聚合型页面
- 面向接口签名、字段摘要、校验值生成

### 6.4 时间工具

页面内分组：

- Unix 时间戳
- GUID

说明：

- `GUID` 虽不属于时间，但都属于高频轻工具，放在这里可减少左侧入口数量
- 如果后续再加随机 ID、雪花算法、日期格式转换，也能自然扩展

### 6.5 结构转换

页面内分组：

- Json To C#
- Json Schema
- Markdown 预览
- MIME 查询，可后续补入

页面角色：

- 中频辅助入口
- 面向结构生成、文档整理和开发支撑

## 7. 迁移实施顺序

### 第一阶段，建立“开发辅助”骨架

- 在 `DbTools` 中新增一级入口 `开发辅助`
- 建立 4 到 5 个二级入口骨架
- 先接入统一的页面容器、标题、说明和状态反馈

### 第二阶段，迁移第一批高频能力

建议顺序：

1. JSON
2. SQL 格式化
3. 编码解码，先 Base64 / URL / Unicode
4. 签名摘要，先 Hash / HMAC
5. 时间工具，先时间戳 / GUID

### 第三阶段，做工作流整合

- 查询结果与 JSON 工具互通
- 文本框内容支持一键送入辅助工具
- 导出场景与 Markdown / 结构转换互通
- 高频工具做最近使用或常用入口

### 第四阶段，决定 AzrngTool 后续定位

可选方向：

- 继续保留为独立实验仓库
- 降级为内部能力验证仓库
- 逐步停止新增功能，只保留迁移来源价值

## 8. 技术迁移建议

### 8.1 迁什么

优先迁移：

- 纯算法逻辑
- 工具服务
- 校验与错误处理逻辑
- 相对独立的 ViewModel 设计思路

### 8.2 不直接迁什么

- `AzrngTool` 的主窗口结构
- 原有工具箱导航模型
- 与 `DbTools` 当前视觉语言不一致的整页布局

### 8.3 目标结构建议

在 `DbTools` 中新增类似目录：

- `ViewModels/DeveloperTools/`
- `Views/DeveloperTools/`
- `Services/DeveloperTools/`
- `Models/DeveloperTools/`

这样能保持数据库主线与辅助能力的边界清晰。

## 9. 风险与注意点

- 不要让 `开发辅助` 反客为主，重新把产品做成大杂烩工具箱
- 不要一次性迁入太多页面，先用第一批验证信息架构是否成立
- 不要直接拼接两套设计系统，统一服从 `DbTools` 的样式体系
- 不要把“数据库主工作区”和“辅助工具工作区”混成一个页面

## 10. 建议的首批落地范围

如果只做一轮最小可验证版本，建议先落：

1. `JSON`
2. `编码解码`
3. `签名摘要`
4. `时间工具`

这样就能快速验证：

- 用户是否愿意在数据库工具里顺手用这些辅助能力
- `方案 B` 的导航是否足够清晰
- 辅助工具是否真的提升数据库开发效率

## 11. 结论

建议采用以下融合方向：

- `DbTools` 做主产品
- `AzrngTool` 做能力来源
- 采用 `方案 B` 的导航组织
- 先迁高频、强相关、低耦合的第一批工具
- 所有 UI 与交互统一服从 `DbTools`

这样既能保住 `DbTools` 的数据库主线，也能吸收 `AzrngTool` 中真正有价值的通用辅助能力。

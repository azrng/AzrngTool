# AGENTS.md

## 适用范围

- 作用域：仓储、数据访问、依赖注入、配置、构建发布与交付验证规则
- 触发场景：涉及 Repository、数据库、依赖注入、配置、启动方式、打包发布时阅读

### 阅读摘要
- 建议阅读：改仓储、改数据访问、改依赖注入、改配置、改发布链路
- 可先跳过：纯界面样式、纯业务编排、仅文档整理
- 优先查看：数据访问 / 配置规则、构建发布规则、`提交前最小回归`

### 常见任务入口
- 改 Repository 或数据库结构：先看数据访问规则与迁移约束
- 改依赖注入、配置、连接串：先看配置规则与运行约束
- 改构建、发布、打包流程：先看交付规则与验证要求
- 补基础设施回归：先看 `提交前最小回归`

## 技术栈

### 数据与配置
- 数据访问：Dapper + SQLite
- 数据库：SQLite（本地默认）/ PostgreSQL（按项目需要扩展）
- 依赖注入：`Microsoft.Extensions.DependencyInjection`

### 发布
- 桌面应用打包：单文件发布
- 版本、发布目录和运行命令必须可追踪

### 依赖注入规范
- 使用 Azrng 标记接口方式批量注册服务：
  - `ITransientDependency`：临时注入
  - `IScopedDependency`：作用域注入
  - `ISingletonDependency`：单例注入
- 在 `Program.cs` 中调用 `services.RegisterBusinessServices(assemblies)` 扫描并注册服务
- 服务类实现对应标记接口，自动被扫描注册到 DI 容器

### 技术选择原则
如果仓库已经有真实实现，以现有代码为准，不要强行重构或替换技术栈。

**技术债务评估框架**：
- **沿用现有实现**：代码可正常运行、无明显性能问题、维护成本可接受、团队熟悉度高
- **必须重构**：存在安全漏洞、严重影响性能、阻碍新功能开发、维护成本过高
- **可选重构**：代码风格不一致、存在更好的替代方案、但不紧急
- **禁止重构**：仅为个人偏好、追赶技术潮流、非关键路径的过度优化

**重构决策流程**：
1. 先评估现有代码的技术债务等级（高 / 中 / 低）。
2. 判断重构的紧急程度和业务价值。
3. 评估重构风险和工作量。
4. 高风险或大规模重构需先与团队确认。

## 推荐目录结构

- 基础设施目录建议聚焦在 `src/AppName/Repositories/`、`src/AppName/Data/`、`src/AppName/Program.cs` 附近组织，优先复用现有结构，不强制迁移。

```text
src/AppName/
├── Repositories/
│   ├── Interfaces/           # 仓储接口定义
│   └── Implementations/      # 仓储实现
├── Data/
│   ├── Context/              # 数据库上下文 / 连接工厂
│   └── Migrations/           # 数据库迁移脚本
├── Helpers/                  # 基础设施工具类
└── Program.cs                # 依赖注入与启动配置
```

## 基础设施规则

### Repository 层规则
- 仓储类必须实现接口，接口定义在 `Repositories/Interfaces/` 目录。
- 仓储实现负责具体数据访问，不把 SQL、连接串或事务细节泄漏到 ViewModel / Service。
- 仓储方法必须优先采用异步形式，并支持取消令牌。
- 涉及事务时，使用 `IDbTransaction` 或等价机制统一管理。
- 仓储类实现 `ITransientDependency` / `IScopedDependency` / `ISingletonDependency` 接口，通过 `RegisterBusinessServices` 批量注册。

### 数据访问规则
- 使用 Dapper 进行数据访问。
- SQL 查询必须使用参数化查询，禁止字符串拼接构造可注入 SQL。
- 数据库连接字符串从配置文件读取，禁止硬编码。
- 数据库连接必须正确管理生命周期。
- 数据库存储字段建议使用 `snake_case`，映射通过 `Column` 特性或显式映射处理。

### 数据库迁移规则
- 涉及数据库结构变更时，必须同步补齐迁移脚本或等价初始化逻辑。
- 迁移脚本应保持可追踪、可重复执行，并尽量说明适用版本。
- 若支持回滚，回滚方式应与正向变更一起说明。
- 不能确认迁移影响范围时，先说明风险，不要直接执行破坏性变更。

### 配置与环境规则
- 运行路径、数据库文件路径、连接串、环境依赖必须通过配置管理，禁止散落在代码常量中。
- 本地默认配置应尽量可直接运行，同时避免把密钥、真实凭据写入仓库。
- 新增配置项、命令或环境依赖时，必须同步更新相关文档。

## 发布规则

### 总体要求
- 优先复用现有打包配置。
- 生产发布优先采用单文件发布（Single File Publish）。
- 不要把密钥写入配置文件或发布产物。
- 发布链路要可追踪。
- 清理旧版本时要保留最小回滚窗口。

### 发布交付规则
- 只要本次任务涉及发布配置、版本号、运行命令或环境依赖，完成实现后应主动执行一次构建验证。
- 发布验证至少包括：
  - 构建是否成功
  - 应用是否成功启动
  - 关键功能是否处于可用状态
  - 核心访问链路是否可用
- 若因环境限制无法执行，必须明确说明。

## 测试与验证规则

### 仓储与数据验证
- Repository 层 SQL 查询、数据映射、事务处理发生变化时，应补充对应测试或最小化验证步骤。
- 涉及数据库初始化、迁移、升级时，应至少验证一次从空环境到可运行状态的链路。
- 不要让测试或验证依赖手工准备的本地状态。

### 无法执行验证时
- 必须说明未执行的验证类型。
- 必须说明未执行原因。
- 必须说明潜在影响范围和风险。


### 提交前最小回归
- 修改 Dockerfile、Compose、启动命令、打包脚本或端口映射时：至少执行一次构建与启动验证
- 修改环境变量、配置项或连接串时：确认示例配置、文档和运行方式同步更新
- 修改数据库、仓储、依赖注入或外部服务集成时：至少补一项真实链路或等价验证
- 若本次改动同时影响应用访问链路，应至少补一次核心链路检查
